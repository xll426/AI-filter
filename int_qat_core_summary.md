# 整数量化感知训练（Int-QAT）方案总结

## 1. 目标与问题定义

本文档总结当前讨论的核心目标、关键概念、训练与部署路径，以及容易混淆的疑难点。目标不是泛泛而谈“量化”，而是围绕一个**特定的整数滤波部署需求**给出完整、准确、工程可落地的说明。

当前目标可以概括为：

- 输入为 **8-bit 像素整数**，语义上对应 `0 ~ 255`。
- 模型核心卷积权重最终需要落为 **不超过 17 bit 的整数**。
- bias 也希望尽量落为 **不超过 17 bit 的整数**。
- 模型输出仍为 **8-bit 像素整数**，即 `0 ~ 255`。
- 中间 MAC 累加器可以比 17 bit 更宽，不强制受 17 bit 限制。
- 希望最终部署到硬件时，只走**原始像素整数输入 → 整数卷积乘加 → 整数缩放/右移 → 残差加回 → 截断到 0~255**这一路径。
- 希望尽量接近已有的 FP32 基线模型效果。

这不是普通的“把网络量化到 int8 大概能跑”问题，而是一个**强约束、面向硬件实现的定点滤波模型训练问题**。

## 2. 当前工程的真实状态

### 2.1 数据侧：训练时输入已经归一化到 `[0,1]`

当前数据集代码中：

- 磁盘上保存的是原始 YUV 数据。
- 训练读取时使用 `normalize=True`。
- 对 8-bit 数据，这等价于：

\[
x_{norm} = \frac{x_{raw}}{255}
\]

因此，当前训练 dataloader 实际输出的张量语义是：

- `input`：`[0,1]`
- `target`：`[0,1]`

这与当前 loss 设置是一致的。例如：

- `MS-SSIM` 默认 `data_range=1.0`
- `PSNR/SSIM` 的计算也按 `[0,1]` 语义来理解

因此，当前训练工程不是直接在 `0~255` 域里训的，而是在 **归一化域** 里训的。

### 2.2 模型侧：当前模型是一个 Y 通道残差滤波器

当前 `PrefilterNet` 的主路径可以概括为：

\[
Y \xrightarrow{\text{PixelUnshuffle}(4)} Y_u
\xrightarrow{\text{processing}} \Delta_u
\xrightarrow{\text{residual add}} Y_u + \Delta_u
\xrightarrow{\text{PixelShuffle}(4)} Y_{out}
\]

其中：

- 只对 Y 通道进行处理。
- UV 通道在 `only_train_y=True` 时基本直接旁路回去。
- `PixelUnshuffle(4)` 会把 1 个 Y 通道重排成 16 个子通道。
- `processing` 是 `MBRConv3` 多分支卷积块。
- `processing.slim()` 可以把多分支训练态结构融合成一个部署态单 `3×3` 卷积核和一个 bias。

因此，部署后核心运算图实际上很规整：

\[
Y_u \xrightarrow{\text{single fused 3x3 conv}} \Delta_u
\xrightarrow{+Y_u} Y_u^{out}
\]

这也是为什么该模型非常适合做定点整数化训练与部署。

## 3. 为什么不能只看“最后结果是整数”

一个容易混淆但极其关键的点是：

> **最后输出是整数，不等于卷积权重可以只用整数且不需要缩放。**

### 3.1 一个基本例子

假设某个残差是：

\[
\Delta = 0.1x_1 + 0.2x_2
\]

若：

\[
x_1 = 100, \quad x_2 = 50
\]

则：

\[
\Delta = 0.1 \times 100 + 0.2 \times 50 = 20
\]

虽然最后结果 `20` 是整数，但：

- 系数 `0.1` 和 `0.2` 并不是整数。
- 最终结果之所以是整数，是因为**小数权重乘整数输入后再相加**，恰好得到了整数或接近整数的结果。

### 3.2 这对滤波器尤其重要

滤波器特别依赖很多**很小的正负小数系数**，例如：

\[
K =
\begin{bmatrix}
0 & 0.05 & 0 \\
0.05 & -0.20 & 0.05 \\
0 & 0.05 & 0
\end{bmatrix}
\]

这种核的特点是：

- 对邻域做轻微加权
- 对中心做轻微压制
- 最终只产生一个小的残差修正

如果不允许权重表示小数，而只允许权重直接取整数，例如：

\[
..., -2, -1, 0, 1, 2, ...
\]

那么滤波器的“力度”就会变得非常粗糙，很难精细地表达小残差。

## 4. 为什么需要缩放（shift / 分母）

### 4.1 缩放的本质不是防中间 MAC 溢出

在当前讨论的设定里，中间 MAC 累加器允许更大位宽，所以：

- 右移或除以 \(2^s\) **不是为了防中间 MAC 溢出才必须存在**。

它的更本质作用是：

> **给整数权重一个“隐含分母”，让整数权重可以表示原本的浮点小数权重。**

### 4.2 定点表达的本质

如果定义：

\[
W_{eff} = \frac{Q}{2^s}
\]

其中：

- \(Q\) 是硬件里存储的整数权重
- \(s\) 是固定整数 shift
- \(W_{eff}\) 是实际等效的小数权重

那么原本的浮点卷积：

\[
\Delta = \sum_i w_i x_i + b
\]

可以用定点形式表示为：

\[
\Delta \approx \frac{\sum_i Q_i x_i + Q_b}{2^s}
\]

这样一来，即使硬件只存整数 `Q_i`，也能表示：

- \(0.05\)
- \(-0.2\)
- \(0.03125\)
- \(-0.0625\)

这类小数滤波器系数。

### 4.3 一个具体数值例子

若：

\[
W = 0.05, \quad s = 8
\]

则：

\[
Q = \text{round}(0.05 \times 256) = 13
\]

对应的定点有效权重为：

\[
W_{eff} = \frac{13}{256} \approx 0.05078
\]

这已经很接近原始 FP32 权重 `0.05`。

因此，所谓“需要缩放”，本质是：

> **为了表示小数滤波器，而不是为了单纯把大数拉回小数。**

---

## 5. 为什么 bias 不能替代缩放

一个常见误区是：

> 既然量化后整数卷积会有误差，那能不能靠 bias 把它拉回来？

答案是：**不能替代。**

### 5.1 原因：bias 是常量项，不是结构响应项

卷积输出是：

\[
\Delta = \sum_i w_i x_i + b
\]

其中：

- \(\sum_i w_i x_i\) 是随输入邻域内容变化的局部结构响应
- \(b\) 是固定常数项

所以 bias 能做的是：

- 所有位置统一加 2
- 所有位置统一减 3

但它做不到：

- 这个边缘位置减 2
- 那个平坦区域不变
- 另一个纹理区域加 1

### 5.2 举例说明

假设理想小数卷积核在某个 patch 上应该给出：

\[
\Delta_{target} = -2.5
\]

而某个粗糙整数核给出的结果是：

\[
\Delta_{int} = -50
\]

若强行加一个 bias：

\[
b = +48
\]

则该 patch 上结果变成：

\[
-50 + 48 = -2
\]

看似“拉回来了”。

但如果换一个平坦区域，原本理想输出应接近：

\[
\Delta_{target} = 0
\]

整数核本来也给：

\[
\Delta_{int} = 0
\]

此时再加同样的 bias `+48`，就会变成：

\[
48
\]

这明显是错误的。

因此：

> **bias 只能补全局常量偏移，无法替代卷积核的小数结构表达能力。**

## 6. 推荐的整数硬件执行形式

对于当前模型，最合理的硬件目标形式是：

### 6.1 原始输入

\[
Y_{raw} \in [0,255]
\]

### 6.2 子像素重排

\[
Y_u = \text{PixelUnshuffle}(Y_{raw})
\]

### 6.3 整数卷积 + 偏置

\[
acc = \text{Conv}(Y_u, Q_w) + Q_b
\]

### 6.4 固定右移 / 定点缩放

\[
\Delta = \text{round}\left(\frac{acc}{2^s}\right)
\]

### 6.5 残差加回并裁剪

\[
Y_u^{out} = \text{clip}(Y_u + \Delta, 0, 255)
\]

### 6.6 重排回图像

\[
Y_{out} = \text{PixelShuffle}(Y_u^{out})
\]

这个形式满足：

- 输入是原始像素整数
- 核心权重和 bias 是整数
- 缩放只在残差输出处做一次固定 shift
- 最终输出仍是原始像素整数

这是一条非常硬件友好的路径。

## 7. 当前 QAT 方案的本质

## 7.1 不是直接训练 int12/int17 张量

当前 `model_int_qat.py` 方案里，真正被优化器更新的是：

- `weight_fp`
- `bias_fp`

它们是浮点主参数（master parameters）。

这并不等于“训练时就按普通 FP32 权重直接卷积”。

### 7.2 真正参与前向的是量化后的整数语义参数

训练时每次 forward 都会执行：

\[
Q_w = \text{clamp}(\text{round}(W_{fp} \cdot 2^s), q_{min}, q_{max})
\]

\[
Q_b = \text{clamp}(\text{round}(b_{fp} \cdot 2^s), b_{min}, b_{max})
\]

然后卷积真正用的是：

\[
acc = \text{Conv}(Y_u, Q_w) + Q_b
\]

再通过：

\[
\Delta = \text{round}(acc / 2^s)
\]

得到残差。

因此，更准确的说法是：

> **不是 int12/int17 参数本体直接被优化器训练，而是模型每一步都按 int12/int17 约束下的前向行为在训练。**

---

## 8. 为什么还需要训练，而不是直接导出就结束

一个重要疑问是：

> 既然已经有训练好的 FP32 模型，直接做
> \(W_{fp} \cdot 2^s\) → round → clamp，
> 不就已经差不多了吗？

这个判断在高 bit、浅网络场景下，**有可能成立一部分**。

### 8.1 什么时候直接导出可能已经够好

当前模型具有以下特点：

- deploy 后非常浅
- 只有一个核心 fused 3×3 卷积
- bit 允许到 12、14、16 甚至 17
- 是局部残差滤波而不是超深大模型

所以直接量化后，可能已经“大差不差”。

### 8.2 但它不是离散整数空间里的真正最优点

FP32 训练得到的是：

> **连续参数空间中的最优点**

而部署时你真正执行的是：

- `round`
- `clamp`
- `shift`
- `clip`

这意味着部署空间是：

> **离散格子 + 有边界的整数空间**

因此：

- 原来的 FP32 最优点
- 并不一定是离散整数空间中的最优点

QAT 的意义不是“神奇消除 round 误差”，而是：

> **在量化前向已经固定的前提下，把参数重新微调到一个更适合整数格子的位置。**

### 8.3 QAT 具体在补什么

QAT 最常补回的内容包括：

1. **把参数推到更稳的量化格子附近**，减少边界抖动。  
2. **重新调整不同权重之间的分工**，补偿量化误差。  
3. **适应 clamp / shift / clip 的存在**，让整个整数执行路径更稳定。  
4. **减少参数长期撞边界**，让最终导出的 `q_w/q_b` 更可用。  

因此：

- PTQ：把 FP32 参数硬投影到整数格子上
- QAT：承认以后只能踩整数格子，再在格子世界里重新优化一下

## 9. STE 的作用与边界

### 9.1 为什么需要 STE

因为当前前向中包含：

- `round`
- `clamp`
- `clip`

这些操作本身不可导或几乎处处导数为 0。  
如果直接原样参与反向，训练几乎无法进行。

STE（Straight-Through Estimator）的思想是：

- 前向仍按真实量化规则走
- 反向把这些离散操作近似看成恒等映射或分段线性映射

例如：

\[
\text{STE-round}(x) = x + (\text{round}(x) - x)_{\text{detach}}
\]

前向值等于 `round(x)`，但反向梯度近似：

\[
\frac{\partial \text{STE-round}(x)}{\partial x} \approx 1
\]

### 9.2 STE 不能做什么

STE **不是魔法**。它不能：

- 恢复已经丢失的表达能力
- 让本来 8bit 表达不了的东西突然无损表示
- 替代合理的 bit 设计和缩放设计

STE 的真正作用是：

> **让你在“整数前向已固定”的前提下，还能继续做梯度优化。**

## 10. 梯度回传到底怎么设计

当前推荐方案中，训练变量不是：

- `q_w`
- `q_b`

而是：

- `weight_fp`
- `bias_fp`

forward 时先做：

\[
latent^{(w)} = weight_{fp} \cdot 2^s
\]

\[
Q_w = \text{STE-round-clamp}(latent^{(w)})
\]

\[
latent^{(b)} = bias_{fp} \cdot 2^s
\]

\[
Q_b = \text{STE-round-clamp}(latent^{(b)})
\]

### 10.1 对权重的梯度

忽略 clamp 饱和的边界影响时，近似有：

\[
\frac{\partial L}{\partial weight_{fp}}
\approx
2^s \cdot \frac{\partial L}{\partial Q_w}
\]

也就是说：

- forward 用的是量化后的整数语义权重
- backward 梯度最终仍然回到 `weight_fp`

### 10.2 对 bias 的梯度

同理：

\[
\frac{\partial L}{\partial bias_{fp}}
\approx
2^s \cdot \frac{\partial L}{\partial Q_b}
\]

### 10.3 对 shift 的处理

第一版推荐：

- `shift` 先固定，不训练

因为：

- `shift` 是离散整数变量
- 直接训练它不稳
- 先固定 shift，只训练 `weight_fp/bias_fp` 就已经能跑通整数约束训练

后续若需要更精细优化，可训练一个连续 scale，再最终逼近为：

\[
\alpha \approx 2^{-s}
\]

## 11. 为什么需要 `L_reg`

这里的 `L_reg` 不是传统意义上的“为了防过拟合”的正则，而是：

> **为了让量化前的整数 latent 不要总想冲出硬件可表示范围。**

### 11.1 范围正则

公式是：

\[
L_{range} = \text{mean}(\text{ReLU}(|x|-q_{max})^2)
\]

其中：

- `x` 是量化前的整数 latent，例如：

\[
x = weight_{fp} \cdot 2^s
\]

- \(q_{max}\) 是当前 bit 允许的最大整数

这个正则的含义是：

- 没超范围：罚 0
- 超出范围：按超出的量平方惩罚

也就是：

> **不是让所有参数都变小，而是只惩罚那些想冲出 bit 范围的参数。**

### 11.2 为什么不能只靠 clamp

因为 clamp 只是前向把它截住：

- 表面上参数没出界
- 但训练中的主参数可能一直想往外跑

结果会导致：

- 参数长期贴边界
- 量化后很多值塌成同一个上限
- 模型表达能力下降
- 导出不稳定

所以 `range_penalty` 的作用是：

> **提前把参数往合法范围里拉，而不是等越界后再硬砍。**

### 11.3 bias L1 的作用

bias L1 不是必须，但有时很有价值。它的作用是：

- 鼓励 bias 本身更小
- 让模型更多依赖卷积核结构，而不是依赖一个大常量去整体抬亮/压暗

这对残差滤波任务尤其合理。

## 12. `q_w_float` 看起来是 float，为什么仍然是整数权重

这是一个非常容易引起误解的点。

在导出推理模块中，经常会看到类似：

```python
self.register_buffer("q_w_float", q_w.to(torch.float32))
```

这里要区分两件事：

### 12.1 dtype 层面

它的存储 dtype 是 `float32`。

### 12.2 数值语义层面

它里面装的值仍然是整数，例如：

- `13.0`
- `-51.0`
- `2047.0`

这不是普通意义上的“浮点小数卷积权重”，而是：

> **用 float32 容器装着的整数常量。**

之所以这么做，通常只是为了：

- 便于 ONNX / PyTorch 图中表达 Conv 常量
- 便于工具链导出

它并不意味着又回到了普通 FP32 权重推理。

## 13. 为什么推理导出时看不到“再乘一次 shift”

因为这一步已经在导出 `q_w/q_b` 时做完了。

训练/导出阶段：

\[
Q_w = \text{round}(W_{fp} \cdot 2^s)
\]

\[
Q_b = \text{round}(b_{fp} \cdot 2^s)
\]

部署推理阶段直接使用：

\[
acc = \text{Conv}(Y_u, Q_w) + Q_b
\]

然后再：

\[
\Delta = \text{round}(acc \cdot 2^{-s})
\]

因此部署时不再显式看到：

\[
W_{fp} \cdot 2^s
\]

是因为：

> **`Q_w` 已经是提前乘完 shift 再 round 后保存好的整数权重。**

所以部署只需要做后半步：

- 再把累加结果按 \(2^{-s}\) 缩回来

这两种写法完全等价。

## 14. 当前推荐的训练路线

推荐的训练路线不是一步到位的“纯 end-to-end 神秘训练”，而是分阶段的。

### 阶段 1：保留已有 FP32 baseline

继续把当前 `PrefilterNet` 当作：

- 初始化来源
- 精度上界参考

### 阶段 2：构建 deploy-FP32 teacher

使用 `processing.slim()` 得到：

- `fused_weight_fp`
- `fused_bias_fp`

构建 deploy 结构的 FP32 teacher：

\[
Y \rightarrow PixelUnshuffle \rightarrow Conv3x3 \rightarrow residual add \rightarrow PixelShuffle
\]

### 阶段 3：构建 deploy-int-QAT student

student 在 forward 中显式模拟：

- `x_norm -> x_raw`
- 整数卷积参数生成
- 右移/shift
- raw 像素残差加回
- clip 到 `[0,255]`
- 再除回 255 接 loss

### 阶段 4：loss 组成

推荐形式：

\[
L = L_{task} + L_{distill} + L_{reg}
\]

其中：

#### 任务损失

\[
L_{task}
\]

继续复用当前：

- ROI Charbonnier
- MS-SSIM
- EdgeConsistency

#### 蒸馏损失

\[
L_{distill}
\]

student 对齐 deploy-FP32 teacher，默认对 Y 通道做 L1 或 Charbonnier。

#### 正则项

\[
L_{reg}
\]

包括：

- 权重范围正则
- bias 范围正则
- 可选 bias L1

## 15. bit 数应该怎么选

### 15.1 不建议一开始“训练 bit 本身”

bit 是离散超参数，直接做可学习 bit-width 会让系统复杂很多。对当前这种 deploy 后非常小的模型来说，第一版更推荐：

- 先固定 bit
- 训练 `weight_fp / bias_fp`
- 搜索几个 bit 档位

### 15.2 推荐优先测试的权重 bit 档位

建议测试：

- 8 bit
- 10 bit
- 12 bit
- 14 bit
- 16 或 17 bit

### 15.3 当前最推荐的起点

最推荐从下面这一组开始：

- **weight = 12 bit**
- **bias = 17 bit**
- **activation = 8 bit 像素语义**
- **per-channel shift**

原因：

- W8 可能已经可以工作，但对细小滤波修正不一定够稳
- W12 往往已经足够细，能更接近 FP32 小数滤波器
- W17 虽然更保险，但可能不是最优性价比点

---

## 16. 什么时候可以不做 QAT，什么时候值得做

### 16.1 可以先直接做 PTQ 验证

既然你已经有高质量 FP32 模型，并且 deploy 结构很浅，那么最现实的第一步应当是：

1. `slim()`
2. 选定 shift
3. 直接导出 `q_w/q_b/shift`
4. 在验证集上看：
   - PSNR
   - SSIM
   - selective_score
   - edge_retention_ratio
   - 视觉效果

### 16.2 如果直接量化后已经几乎不掉点

那说明：

- 高 bit 定点化已经足够好
- QAT 不是必须
- 或只需要很短的微调

### 16.3 如果直接量化后掉了一点

这就是最适合做短程 QAT 微调的情况。因为：

- 模型结构已经固定
- 量化规则已经固定
- 只需要在整数格子上重新挪一挪参数

### 16.4 如果直接量化后掉很多

要先判断原因：

- bit 太低？
- bias 限制太死？
- shift 初始化太差？
- 是否大量参数撞边界？

如果这些问题本身就很严重，单靠 QAT 也不可能神奇恢复全部损失。

---

## 17. 最终导出与验证原则

最终导出的不是：

- `weight_fp`
- `bias_fp`

而是：

- `q_w`
- `q_b`
- `shift`

并建议至少验证三条路径：

1. 原训练态 FP32 模型  
2. deploy-FP32 reference  
3. deploy-int-QAT student  

理想关系应当是：

- 1 和 2 差距很小，说明 `slim()` 正确
- 2 和 3 差距尽量小，说明整数化损失可控
- 3 和最终导出 reference int 推理一致，说明部署链路闭环正确

---

## 18. 当前方案的最准确一句话总结

当前推荐方案可以最准确地概括为：

> **float master params + int-constrained forward + final integer export**

也就是：

- 训练时，优化器更新 float 主参数，保证可训练性
- 每次 forward，都强制把参数投影到有限 bit 的整数语义下执行，保证与部署一致
- 最终导出 `q_w / q_b / shift`，保证真正能落到硬件

这不是“普通 FP32 训练后最后随手量化一下”，也不是“直接拿 int12 参数做反向传播”。

它是：

> **在保持可训练性的同时，让前向行为尽可能贴近最终的整数硬件执行路径。**

---

## 19. 最终建议

对于当前这个任务，推荐实际操作顺序如下：

1. **保留现有 FP32 baseline**，作为精度参考。  
2. **先做直接导出 + 验证**，看高 bit 定点是否已经足够接近。  
3. 若直接导出效果已接近，可只做很短的 QAT 微调。  
4. 若仍有明显损失，则做完整的 deploy-int-QAT 微调：  
   - `L_task + L_distill + L_reg`  
   - 权重 12 bit 起步  
   - bias 17 bit 目标  
   - per-channel shift  
5. 最终导出 `q_w/q_b/shift`，并做 bit-true 验证。  

在当前模型结构、bit 上限和任务性质下，**最值得优先尝试的点**是：

- **W12 + bias17 + per-channel shift + 短程 QAT 微调**

这是当前最平衡、最有希望接近 FP32 的路线。
